The Cheap-Then-Clean Architecture

A two-pass pattern for continuous AI pipelines operating under strict local compute budgets.

Adapted from a January 2025 Pieces conversation and developed in the Research Archivum. Edited for clarity and factual precision; claims tied to deployment work remain at the level already discussed publicly.

Idea
Uniformly spending expensive compute on a repetitive stream wastes most of the budget.
Mechanism
Run a cheap first pass continuously; consolidate and clean only when enough new signal has accumulated.
Implication
Resource-rational pipelines can improve both quality and efficiency by deciding when expensive processing is deserved.

Why two imperfect passes beat one expensive pass

Here's a common problem in local AI: you need to process something continuously (like screen content), but the best processing is expensive, and you can't afford to run it on every frame.

The naive solution is to find a cheaper model. Accept lower quality for better performance.

A better solution is a two-pass architecture: cheap extraction followed by expensive cleanup. You get the speed of a fast model AND the quality of a slow model, by using each where it makes sense.

The Screen Capture Problem

To build a long-term memory system, you need to understand what the user is looking at. That means capturing screen content and extracting meaningful text.

The challenge: screens change constantly. New windows open, documents scroll, applications switch. You need to capture frequently—multiple times per second—to not miss important content.

But good OCR is expensive. A high-quality vision model might take hundreds of milliseconds per frame. At that speed, you can't keep up with real-time screen changes, and you're burning CPU cycles that the user needs for their actual work.

The Two-Pass Solution

The public technique is a two-pass pipeline:

Pass 1: Fast and Cheap
A lightweight OCR model runs frequently. Its exact latency depends on the model, capture rate, hardware, and runtime. It's also imperfect. The output has errors: garbled characters, missed words, confused layouts.

But that's okay. We're not trying to get perfect text. We're trying to get something we can work with.

Pass 2: Slow and Thorough
After a suitable interval, a heavier model processes the accumulated captures. The interval is deployment-dependent and should be selected against latency and compute budgets. This model:

  • Receives the noisy text from Pass 1
  • Cleans up OCR errors
  • Reconstructs coherent narratives
  • Extracts meaningful structure

The second pass is deliberately more expensive, but it runs less often; the relevant quantity is measured end-to-end cost per useful unit of information.

Why This Beats Single-Pass

Consider the alternatives:

Option A: Run expensive OCR on every frame
Quality: High. Cost: Astronomical. Your laptop becomes a space heater.

Option B: Run cheap OCR on every frame, use directly
Quality: Low. Cost: Low. But the errors compound—garbage in, garbage out.

Option C: Run cheap OCR frequently, expensive cleanup occasionally
Quality: Potentially higher after cleanup. Cost: Moderate. The result still has to be measured.

The key insight: most frames don't contain new information. The screen often shows the same content for seconds at a time. Running expensive processing on every frame is wasteful—you're re-processing the same content repeatedly.

By batching cleanup, you process each piece of content once at high quality, rather than many times at low quality.

The Cleanup Model

What does “cleanup” actually mean? A cleanup stage can perform several operations:

Error correction: "teh" → "the", "l0cal" → "local". Common OCR mistakes have patterns that can be learned.

Layout reconstruction: Raw OCR often loses paragraph structure, confuses columns, interleaves headers with body text. The cleanup model reconstructs logical reading order.

Deduplication: If the same content appears in multiple captures (because the screen didn't change), keep one copy.

Narrative extraction: Beyond just cleaning text, identify what the user was actually doing. "User was reading documentation about API authentication" is more useful than raw text.

Resource Budgeting

The two-pass architecture is really about resource budgeting. You have a limited compute budget—the user's CPU cycles that you can consume without degrading their experience.

How do you spend that budget? You could spend it uniformly: moderate processing on every frame. Or you could spend it strategically: minimal processing most of the time, intensive processing when it matters.

Strategic spending can win when the cheap pass reliably identifies where the expensive pass adds value. The uniform approach wastes resources on frames that don't need them and skimps on frames that do.

Generalizing the Pattern

This cheap-then-clean pattern applies beyond OCR:

Audio processing: Fast voice activity detection, slow transcription only when speech is detected.

Video analysis: Fast motion detection, slow scene understanding only when something interesting happens.

Code analysis: Fast syntax highlighting, slow semantic analysis in the background.

Document processing: Fast text extraction, slow summarization when documents are complete.

The principle is the same: use cheap filters to identify what deserves expensive processing, then spend your budget on what matters.

The Pipeline Mindset

Thinking in pipelines changes how you approach AI system design.

Instead of asking "what's the best model for this task?" you ask "what's the best sequence of models?"

The answer is often: a fast, cheap model to filter and pre-process, followed by a slow, powerful model to do the real work. You get efficiency AND quality by using each model where it excels.
