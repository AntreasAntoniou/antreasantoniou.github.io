Research Archivum · Project-planning note

The Context Re-entry Problem

The expensive part of interrupted work is often not the interruption. It is reconstructing the mind that knew what to do next.

Maturity: Design proposal

Developed from an Archivum project-planning note for a context-restoration tool. The source records an initiated idea and proposed workflow only; no integration, demo, or measured productivity result is claimed here.

Problem
Resuming work requires reconstructing goals, decisions, evidence, and unresolved uncertainty.
Mechanism
Generate a small, cited context packet with explicit freshness and invalidation rules.
Test
Measure time and error to the first correct next action, not how fluent the summary sounds.

You reopen a project after eleven days. The editor restores the right files. The task manager shows seven open items. A note says “try the narrower version first,” but not what was narrowed or why. For twenty minutes you are technically working and producing nothing. You are rebuilding the person who understood the project.

That interval is the context re-entry problem. It is often described as the cost of context switching, as though the mind were a CPU paying a fixed penalty each time it changes process. The more useful framing is a reconstruction tax. Serious work has a temporary internal model: the objective, the current frontier, rejected paths, strange local vocabulary, fragile assumptions, and the next experiment that makes sense only because of all the above. When the model decays, reopening the files does not restore it.

A planning-stage Archivum note proposed a tool for producing rapid context refreshes from selected materials. The stronger thesis is broader than any particular service: context re-entry can be designed for, measured, and reduced—but not replaced by magical memory. The system should help a person reconstruct a warranted state, not manufacture the feeling of familiarity.

The tax is paid in wrong starts

Time-to-resume is the obvious cost, but it is not the most damaging one. A person can begin quickly and begin incorrectly. They rerun an experiment already ruled out, reopen a settled design question, treat an old plan as current, or optimise a metric that ceased to matter after the last meeting. Fluent motion disguises stale context.

This suggests a better unit of measurement: time to the first correct next action. Start the clock when the dormant project is opened. Stop it when the person can state the current objective, identify the live uncertainty, explain why the next action follows, and point to the evidence. Record wrong turns and source lookups along the way. The point is not to grade the worker. It is to compare re-entry methods on the same person and project.

The reconstruction tax will vary. A two-day pause in a well-documented task may need one minute. A month away from exploratory research may need an hour because the missing context includes tacit judgments that were never written. A useful tool must admit this variance. Promising “instant context” would merely turn a difficult measurement problem into marketing.

A context packet is a temporary bridge

The proposed mechanism begins with scope chosen by the user: a project folder, a paper, selected notes, perhaps a task log. The system does not vacuum an entire digital life and guess the topic. It processes the bounded material against a specific prompt such as, “Help me resume the evaluation design and decide what to run next.” The output is a context packet.

A good packet contains at least six parts. First, the current objective in operational language. Second, the latest known state: what exists, what was attempted, and what changed. Third, a decision trail including rejected alternatives and reasons. Fourth, open loops—questions, dependencies, and promises. Fifth, a proposed next action explicitly labelled as a proposal. Sixth, provenance links back to the passages from which every important claim was derived.

The packet should be small enough to read in a few minutes and structured enough to interrogate. It is not a canonical project summary. It is a temporary bridge built for one return at one moment. After crossing it, the person should work from the underlying project, not keep asking the bridge what the landscape looks like.

Provenance prevents synthetic déjà vu

Generated summaries are very good at producing recognition: “yes, that sounds like my project.” Recognition is not recall, and neither is truth. A plausible sentence assembled from three stale notes can feel exactly right while reversing the actual decision.

For that reason, provenance must be visible at sentence or claim level for decisions, results, and next steps. If the packet says the team chose the narrower evaluation, it should show the note or transcript segment where that happened. If the source only contains a suggestion, the packet must retain the modal: “considered,” not “decided.” If two records conflict, the packet should expose the conflict rather than selecting the smoother narrative.

This also gives the reader a useful interaction. They can expand the evidence for the one claim that surprises them while accepting low-risk orientation at a glance. The language model becomes a guide through the record, not an oracle above it.

A context packet needs a privacy perimeter

Selecting a record does not settle the right to process it. A project folder may contain comments from collaborators, meeting notes about people who did not choose the tool, or shared records governed by expectations different from the selector’s own notes. Consent therefore has to cover the people and material represented, not merely the person pressing “generate.” Sources without a clear processing basis should stay outside the packet.

The planning design must also make location explicit before ingestion: does the selected material remain on the device, remain inside the organisation, or leave the device and leave the organisation for an external processor? Those are materially different privacy boundaries. The interface should state which applies for this packet, minimise what is sent, and avoid silent fallback from local or organisational processing to external processing. A service identity should have least-privilege, read-only access to the selected scope rather than broad access to an account or archive, and the packet should preserve rather than flatten the source records’ access restrictions.

Retention and deletion need equally concrete rules. Temporary inputs, caches, derived indexes, and generated packets should have declared lifetimes; they should not become training material or a permanent secondary archive by default. Deletion should cover the packet and its derived processing artifacts, with honest disclosure where backups or downstream copies prevent immediate erasure. If bounded retention and meaningful deletion cannot be implemented for a source class, the planning proposal should treat that material as ineligible rather than quietly accepting the risk.

Freshness needs an invalidation model

Most context tools focus on retrieval and ignore decay. Yet a context packet can become wrong five minutes after generation if a new result lands or a decision is reversed. Every packet therefore needs an “as of” time, a manifest of sources consulted, and invalidation triggers.

Some triggers are mechanical: a tracked file changed, a new meeting note entered the scope, an experiment completed, or a task was closed. Others are semantic and harder: a result undermined an assumption or a conversation superseded a decision without using the same words. The planning proposal does not solve this. A first version could be conservative: mark the packet stale whenever scoped material changes and require regeneration. That creates extra work, but it is safer than pretending to understand every consequence.

It should also preserve what it did not see. “No result file was included” is useful context. “The meeting notes end on 4 June” is a warning. Absence made explicit prevents the summary from masquerading as a complete memory.

Where the design should fail honestly

The first proof of concept should not attempt general productivity. Choose several paused projects, create packets from deliberately bounded records, and compare unaided re-entry with packet-assisted re-entry. Measure time to the first correct next action, incorrect assumptions voiced, redundant work proposed, evidence lookups, and the user’s calibrated confidence. A confident but wrong return is worse than a slow, cautious one.

The design should be rejected or narrowed if packets routinely omit the reason behind decisions; if users follow fabricated next steps; if source inspection takes as long as manual reconstruction; if stale packets are hard to detect; or if collecting the necessary material creates more maintenance than it saves. It should also fail gracefully when the decisive context was never recorded. No extraction system can recover a thought that existed only in somebody’s head.

External memory earns its place when it leaves enough trustworthy structure for a future self to think again, not when it claims to remember on our behalf. The goal is not to return to the exact mind that left the project—that mind is gone. The goal is to reach the next warranted action without first reenacting every confusion that produced it.
